## 核心论点

**瓶颈没消失,只是转移了**:写代码不再是瓶颈,转移到 **验证 / 代码评审 / 跨职能协作 / 安全**。

> 旧成本结构(写代码贵)→ 旧流程(规划 / 评审 / 会议为主)
> 新成本结构(token 极便宜)→ 整套上下游流程需重新设计

## 5 个失效领域 + 5 条新规范

### 1. 规划方式: JIT 规划

- 砍掉 "写代码前必须先写设计文档" 的仪式 (Fiona 称为 *theater*)
- 改为 **原型先行,文档后补**
- 6 个月路线图通常 3 个月就过时
- "Building is cheap, Arguing is expensive" — 让 Claude 把两种方案做成原型,看实物决断

### 2. 自动化: 肌肉记忆

- 每遇到重复工作,条件反射地问 "能不能自动化"
- Fiona 自己案例: 每日手动汇总客户反馈 → 后台自动任务
- 关键不是某个具体自动化,而是 **团队级肌肉记忆**

> 与卡兹克自身实践一致: "一件事重复 3 遍以上,想办法用 AI 自动化掉" —— 不是建议,是要求

### 3. 代码评审: Trust but verify

- Claude 接管 60-70% 的 review 工作量(style / lint / PR 反馈 / bug / 测试补全)
- 人类专攻: 法律合规 / 信任边界 / 产品方向 / 品味判断
- **动态边界**: 今天人做的事,下个模型可能就能做,必须不断重评

### 4. 团队角色: 模糊化

- PM 大量写代码,工程师做内容设计
- 跨职能 gap 从瓶颈变协作器
- 招聘看两类人:
  - 有产品 sense 的 **创意 builder** —— *"Taste is scarce, typing is not"*
  - 有深厚系统背景的工程师(handle "subtly wrong is still wrong")

### 5. 流程主动删除

- 三条管理原则:
  1. 团队保持扁平 + 灵活流动
  2. Claude 能做的就让 Claude 做
  3. **人不会主动删流程,必须有人指名道姓说"这个流程可以走了"**
- Fiona 案例: 一个 weekly review 会,所有人都在看电脑,问一句"我们为啥还开这个会"→ 当场取消
